## Context

当前账号选择入口主要集中在 `src/ui.js` 的 `selectAccount` 与 `selectMultipleAccounts`，底层已经依赖 `@inquirer/prompts`，因此具备基础的方向键导航能力；但账号展示顺序仍沿用注册表中的静态顺序，未体现近期使用偏好。与此同时，`src/registry.js` 目前只保存账号基本信息和活跃指针，没有任何使用历史字段，无法支撑“最近/高频优先”的排序。

这次变更横跨交互层、注册表持久化结构和切换成功后的状态更新逻辑，适合先把排序规则和数据模型定清楚，再进入实现。

## Goals / Non-Goals

**Goals:**
- 让交互式账号选择继续以键盘导航为主，用户可直接通过上下方向键浏览和确认。
- 为账号记录补充最近使用时间与累计使用次数，以支持统一的选择优先级排序。
- 让所有“选择账号”的交互入口复用同一套排序逻辑，减少行为不一致。
- 保持现有按别名/ID/索引的命令行参数直达能力不变。

**Non-Goals:**
- 不修改账号切换的安全收敛、provider 管理和凭据写入语义。
- 不引入复杂的个性化推荐或跨设备同步历史。
- 不改变非账号选择类 prompt 的展示规则。

## Decisions

### 1. 将使用元数据直接存入 registry 的账号对象

为每个账号增加可选字段 `last_used_at` 和 `use_count`，并在切换成功后更新。

选择这个方案的原因：
- 与现有 `registry.accounts[]` 结构兼容，读写路径最短。
- 老数据天然向后兼容，缺少字段时可按“未使用过”处理。
- 测试和调试成本低，不需要额外的历史文件或索引结构。

备选方案：
- 独立维护 `usage.json`。被放弃，因为会引入双文件一致性问题，且对当前 CLI 规模来说没有必要。

### 2. 使用统一的稳定排序函数为交互式列表排序

新增统一排序逻辑，规则为：
- 有使用历史的账号优先于无使用历史的账号。
- 在有使用历史的账号中，按 `last_used_at` 倒序排序。
- 若最近使用时间相同或缺失，则按 `use_count` 倒序排序。
- 若仍无法区分，则回退到当前注册表顺序，保持结果稳定。

选择这个方案的原因：
- 规则明确、易测试，能够同时覆盖“最近使用”和“高频使用”。
- 稳定回退可以避免列表在无历史账号场景下随机抖动。
- `switch`、`remove` 以及后续可能新增的账号选择入口都可以复用。

备选方案：
- 采用加权分数公式。被放弃，因为解释成本高，且在 CLI 场景下不如显式优先级易于验证。

### 3. 不新增专门的导航实现，继续复用 Inquirer 选择组件

交互式单选继续使用 `select`，多选继续使用 `checkbox`，通过规格明确这些入口必须保持键盘可导航，而不是退回到手输序号模式。

选择这个方案的原因：
- 当前依赖已支持上下键导航，无需重复造轮子。
- 复用现有组件可以把改动集中在排序和元数据维护上。

备选方案：
- 自建终端键盘事件处理。被放弃，因为复杂度高，且没有额外收益。

## Risks / Trade-offs

- [历史字段与老注册表并存] → 缺少 `last_used_at` / `use_count` 时按未使用处理，避免迁移阻塞。
- [排序规则与用户直觉不完全一致] → 采用显式、稳定的优先级规则，并通过测试固定行为。
- [仅在切换成功后更新使用历史] → 保持“真实使用”语义，避免因为用户只浏览列表就污染排序。

## Migration Plan

- 该变更不需要离线迁移脚本。
- 现有 `registry.json` 在首次成功切换账号后会逐步补齐 `last_used_at` 和 `use_count`。
- 若回滚代码版本，新增字段只会被旧版本忽略，不影响账号基本读写。

## Open Questions

- 当前默认不将 `list` 命令输出顺序纳入本次规格，只约束交互式选择列表；如果后续希望“展示顺序”和“选择顺序”完全一致，可在实现阶段一并复用排序函数。
