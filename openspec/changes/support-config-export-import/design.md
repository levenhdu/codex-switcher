## Context

当前仓库已经能管理多账号注册表、账号目录中的 `auth.json` 快照，以及工具受控的 live 配置切换，但缺少“把整套已保存配置带走”的路径。现有 `import` 命令只覆盖单个 `auth.json` 导入，不适合迁移多个账号，也无法恢复账号元数据、使用历史和本地账号目录结构。

这个需求同时涉及 CLI 命令设计、配置包数据模型和导入安全边界，因此需要先明确哪些内容允许导出、导入会不会影响 live 状态，以及冲突时如何处理。

## Goals / Non-Goals

**Goals:**
- 支持把工具管理的账号配置导出为单个可移植文件。
- 支持从导出文件恢复账号注册表和账号目录中的认证快照。
- 在导入前完成结构校验和版本校验，拒绝坏文件或不兼容文件。
- 保证导入不会直接覆盖当前 live `auth.json` 或 `config.toml`。

**Non-Goals:**
- 不导出用户原生的整个 `~/.codex/config.toml`。
- 不在本次变更中支持复杂的冲突合并策略或交互式逐条 merge。
- 不自动在导入后切换到某个账号。

## Decisions

### 1. 使用单文件 JSON 配置包承载导出内容

导出文件采用 JSON，包含：
- `version`：配置包版本号
- `exported_at`：导出时间
- `accounts`：账号列表，每项包含账号元数据、使用历史和账号目录中的认证快照内容

选择这个方案的原因：
- 与现有代码栈兼容，无需引入压缩或归档依赖。
- 可读、可测、易于做版本校验。
- 对当前工具规模来说，单文件最便于备份和跨机器传输。

备选方案：
- 使用 zip/tar 包。被放弃，因为会增加依赖和文件处理复杂度，而当前导出内容主要是结构化文本。

### 2. 导入只恢复“已保存账号”，不触碰 live 状态

导入成功后恢复 registry 中的账号记录和账号目录下的认证快照，但不直接写 live `auth.json`、不重置 live `config.toml`，也不自动切换当前账号。

选择这个方案的原因：
- 避免导入操作意外改变用户当前会话。
- 与现有安全收敛规格一致，减少把“离线配置恢复”误变成“在线状态切换”的风险。

备选方案：
- 导入后自动激活导出时的当前账号。被放弃，因为这会把配置恢复和 live 切换耦合在一起，风险过高。

### 3. 导入默认拒绝冲突账号 ID，采用先校验后写入

导入时若发现配置包中账号 ID 与本地已有账号冲突，则整个导入失败；只有在全部结构、版本和冲突检查通过后，才开始写入账号目录和注册表。

选择这个方案的原因：
- 比“静默覆盖已有账号”更安全。
- 能避免半成功导入导致的状态混乱。
- 首版实现范围明确，不需要设计复杂覆盖策略。

备选方案：
- 允许默认覆盖同名账号。被放弃，因为覆盖凭据和元数据属于高风险操作，应在后续单独设计。

## Risks / Trade-offs

- [导出文件包含敏感凭据] → 明确导出文件应视为敏感文件；CLI 文案需提醒用户妥善保管。
- [默认拒绝冲突会增加一次性迁移成本] → 先保证安全；后续如果需要覆盖模式，可独立加变更。
- [不导出整个 live config 可能让部分用户觉得“不完整”] → 通过规格明确工具只迁移自己管理的账号配置，live 配置仍可由切换流程重建。

## Migration Plan

- 不需要对现有本地数据做离线迁移。
- 实现后新增 CLI 命令用于导出与导入配置包。
- 老版本 registry 若缺少使用历史字段，导出时按当前已解析结果原样序列化即可。

## Open Questions

- 当前默认采用 JSON 明文配置包；如果后续用户需要加密导出，可再单独提出变更，不并入本次范围。
